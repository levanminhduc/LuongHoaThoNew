### ISSUE-1: Fetch audit still has false negatives
- Category: scope
- Severity: high
- Problem: `audit-fetch.sh` only catches literal `fetch('/api...')` / backtick variants, not `const url = '/api/...'; fetch(url)`. The plan also audits only `app` + `components`, missing client-side fetches in `lib/hooks` / stores.
- Why it matters: The final “0 direct fetch” bar can pass while in-scope admin/dashboard fetches remain, especially existing `fetch(url)` patterns in dashboards/admin pages.
- Suggested fix: Make the audit flag all standalone `\bfetch\s*\(` in client roots, then classify/allowlist intentional non-data cases. Add smoke fixtures for `fetch(url)` and include `lib/hooks` at minimum.

### ISSUE-2: `useLogout` violates endpoint-constant gate
- Category: sequencing
- Severity: high
- Problem: Task 1.5b creates `lib/hooks/use-logout.ts` with `apiClient.post('/api/admin/logout', {})`, but `ENDPOINTS` is only created later in Task 3.0. The later hard gate `rg -n "['\"]/api/" lib/hooks/` will fail on this W1 hook.
- Why it matters: The plan cannot pass its own W3 endpoint gate without an unplanned cleanup.
- Suggested fix: Move `lib/api/endpoints.ts` creation earlier before `useLogout`, or add an explicit post-Task-3.0 step to migrate `useLogout` to `ENDPOINTS.auth.logout` before running the gate.

### ISSUE-3: 401 storage-clearing test still misses `employee_user`
- Category: correctness
- Severity: medium
- Problem: The Task 1.3 401 test sets/asserts `admin_token`, `auth_token`, `user_info`, and `admin_user`, but not `employee_user`.
- Why it matters: Round 2’s accepted fix said all 5 auth keys are verified; the current test would not catch a regression that leaks `employee_user` across auth boundaries.
- Suggested fix: Add `localStorage.setItem('employee_user', ...)` and assert it is cleared. Also update the expected client test count from 6 to the actual 8 tests.

### VERDICT
- Status: REVISE
- Reason: Remaining audit and sequencing gaps can let direct fetches survive or make the plan fail its own gates.
